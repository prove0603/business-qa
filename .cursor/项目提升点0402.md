## 一、项目概况

- **sentinel-ai**：一个 SQL 风险扫描与 AI 分析平台。扫描 Java 项目中的所有 SQL（Mapper XML、注解 SQL、QueryWrapper），结合表结构 DDL + RAG 业务知识，利用多模型轮询 AI 判断是否存在慢 SQL 风险，并给出优化建议。同时作为 MCP Server 对外暴露 SQL 分析工具。
- **business-qa**：一个基于 RAG 的业务智能问答助手。支持文档上传、分块向量化、多轮对话、SSE 流式输出，并通过 MCP Client 连接 sentinel-ai 获取 SQL 分析能力。

两个项目已覆盖的 AI 能力：**LLM 对话、RAG（向量检索增强生成）、MCP（工具调用/Function Calling）、结构化输出、多模型轮询与 fallback、Embedding 向量化、SSE 流式**。

## 二、还能新加入哪些 AI 方面的功能

你目前的 AI 能力栈已经覆盖了"对话 + 检索增强 + 工具调用"三大核心，这在行业内属于扎实的基础。以下是按**实用价值和学习收益**排序的扩展方向：

### 高价值、直接可落地

| 方向 | 说明 | 在你项目中的落地场景 |
|------|------|---------------------|
| **Agent / 多步推理** | 让 AI 自主规划和执行多步任务，而不是单次 prompt→response | sentinel-ai 中：AI 发现慢 SQL → 自动查索引统计 → 自动生成 ALTER 语句 → 提交 PR。business-qa 中：用户问一个复杂问题 → AI 分解为多个子问题 → 分别检索 → 汇总回答 |
| **评估与对齐（Evaluation）** | 对 AI 输出做自动质量评估、打分 | 你的 SQL 分析结果目前完全依赖 AI 判断，可以加一个"AI 裁判"做二次校验，或者用 eval 框架跑历史数据回测准确率 |
| **Prompt 工程可视化 / A/B 测试** | 将 prompt 模板化、版本化，支持在线修改和效果对比 | 目前 prompt 硬编码在代码里，可以做成可配置的 prompt 模板 + 效果对比面板 |
| **多模态（图片/文档理解）** | 让 AI 理解截图、架构图、ERD 图 | business-qa 中：上传一张数据库 ER 图，AI 直接理解表关系回答问题 |
| **Knowledge Graph（知识图谱）** | RAG 的进化版，将文档中的实体和关系抽取成图谱 | business-qa 中：从催收业务文档中提取"案件→客户→策略→任务"关系图谱，GraphRAG 比纯向量检索在复杂关联问题上效果更好 |

### 中等价值、技术深度好

| 方向 | 说明 |
|------|------|
| **Guardrails（护栏机制）** | 对用户输入做意图分类和安全检查，对 AI 输出做格式验证、敏感信息过滤、幻觉检测。Spring AI 的 Advisor 链天然支持这个 |
| **流式 Agent + 工具调用** | 目前 MCP 调用是同步的，可以做流式 Agent——一边思考一边输出，调用工具时实时显示中间状态 |
| **Fine-tuning / 领域微调数据准备** | 把历史的 SQL 分析结果（输入 SQL + 正确输出）整理为微调数据集，用于训练领域专属模型 |
| **Observability（AI 可观测性）** | 记录每次 AI 调用的 prompt、response、token 用量、延迟、评分，做成可视化 dashboard。你已经有了 `AiLoggingAdvisor`，可以扩展为完整的 trace 链路 |
| **缓存策略升级（Semantic Cache）** | 目前用 SQL hash 做精确缓存，可以升级为"语义缓存"——相似的问题命中已有缓存，减少 API 调用 |

### 前沿探索

| 方向 | 说明 |
|------|------|
| **Code Generation（代码生成）** | sentinel-ai 发现问题 SQL 后，直接生成修复代码（含索引 DDL + 改写后的 Mapper），通过 MCP 工具提交到 Git |
| **Self-RAG / Adaptive RAG** | AI 自主判断"是否需要检索"以及"检索结果是否足够"，减少不必要的向量查询 |
| **Multi-Agent 协作** | 多个 Agent 各司其职（SQL 扫描 Agent、DDL 分析 Agent、优化建议 Agent），协同完成复杂分析任务 |

## 三、对标商业化产品的改进点

对标同类产品（如 Dify、FastGPT、MaxKB 等 RAG 平台，以及 PingCAP 的 SQL 分析工具），你的项目在以下方面有提升空间：

### RAG 质量与效果

| 改进点 | 现状 | 商业产品做法 |
|--------|------|-------------|
| **分块策略** | 按 Markdown 标题 + 段落，最大 1000 字符 | 商业产品支持多种分块策略（按语义、按 token、递归分割），且支持用户调参和预览分块效果 |
| **检索精度** | 单路向量检索，相似度阈值 0.5 | 商业产品做 **Hybrid Search**（向量 + BM25 关键词混合检索）+ **Rerank 重排序**（用 Cross-Encoder 对检索结果二次排序），效果提升明显 |
| **引用溯源** | 有引用来源 | 可以做到段落级高亮引用，用户点击引用跳转到原文档对应位置 |
| **检索效果可视化** | 无 | 提供"调试模式"，展示每次检索命中了哪些分块、相似度分数是多少，方便调优 |
| **多轮对话上下文** | InMemory 对话记忆 | 可持久化对话历史，支持对话摘要压缩（防止长对话 token 爆炸），支持对话分享 |

### 工程化与可靠性

| 改进点 | 说明 |
|--------|------|
| **重试与限流** | AI 调用失败的重试策略更精细（指数退避、按错误类型区分），全局限流防止 API 配额耗尽 |
| **异步任务管理** | 向量化、扫描等异步任务需要进度条、可取消、失败重试、历史记录。目前只有 `@Async` |
| **AI 响应质量兜底** | 结构化输出解析失败时的降级策略（重试 prompt、换模型、返回原始文本） |
| **版本管理** | 知识库文档的版本历史、对比、回滚（你的 TODO 里已经列了） |
| **批量操作** | 批量导入文档、批量向量化、批量分析 SQL |

### 用户体验

| 改进点 | 说明 |
|--------|------|
| **对话体验** | 支持"追问建议"（AI 回答后推荐相关问题）、消息编辑重发、分支对话 |
| **反馈闭环** | 用户对 AI 回答点"赞/踩"，数据用于评估模型效果和优化 prompt |
| **数据看板** | token 消耗统计、模型使用分布、热门问题 TOP N、RAG 命中率、平均响应时间 |
| **Webhook / 通知** | SQL 扫描完成通知、高风险 SQL 告警推送（企微/钉钉/邮件） |
| **API 开放** | 提供 OpenAPI 规范的对外接口，让其他系统集成你的 AI 能力 |

### sentinel-ai 专项

| 改进点 | 说明 |
|--------|------|
| **CI/CD 集成** | 作为 CI Pipeline 的一个 step（类似 SonarQube），PR 合并前自动扫描新增 SQL |
| **EXPLAIN 实际执行** | 目前是 AI 推测是否走索引，商业产品会连接只读从库实际执行 `EXPLAIN ANALYZE`，拿到真实执行计划 |
| **趋势分析** | 同一条 SQL 在不同版本中风险等级的变化趋势，关联数据量增长曲线 |
| **修复跟踪** | 从"发现风险"到"修复上线"的全链路跟踪，含 Git commit 关联 |

## 四、有必要锻炼手写 AI 相关代码的能力吗

**简短回答：有必要，但不需要从零手写所有东西。**

展开说几个层面：

### 1. 你需要理解的核心知识（必须掌握）

这些是"即使 AI 帮你写代码，你也必须能做出正确技术决策"的基础：

- **Prompt Engineering**：这是目前 AI 应用中 ROI 最高的技能。理解 system prompt / user prompt 的设计原则、few-shot、chain-of-thought，以及如何调试和优化 prompt。这个 AI 很难替你做好，因为它需要领域知识和反复实验。
- **RAG 管道的每个环节**：分块策略的选择依据、Embedding 模型的特性和局限、向量检索的 precision/recall 权衡、rerank 的必要性。你需要知道"为什么检索效果不好"的排查路径。
- **AI 应用架构设计**：什么时候用 RAG 什么时候用 Fine-tuning，什么时候 Agent 比直接 prompt 更合适，工具调用链的编排逻辑——这些是架构层面的判断，需要自己拿主意。
- **Spring AI 的抽象模型**：ChatClient / Advisor 链 / ToolCallbackProvider / EmbeddingModel 的设计思想，知道扩展点在哪，出问题知道去哪层排查。

### 2. 你不必执着于手写的部分

- 具体的 CRUD 代码、前端页面、配置类——这些 AI 生成效率确实高，手写没有额外学习价值。
- Embedding 模型的适配器实现——理解接口契约即可，胶水代码让 AI 写。
- 向量数据库的建表和索引——follow 文档和最佳实践即可。

### 3. 建议的锻炼方式

与其"手写代码"，不如聚焦在这几件事：

1. **能不看代码就画出完整的数据流**：用户提问 → ChatClient → Advisor 链 → RAG 检索 → MCP 工具调用 → LLM → 结构化解析 → SSE 推送。你应该能闭着眼讲清每一步。
2. **能独立排查 AI 相关问题**：检索不准、输出格式错误、token 超限、多模型 fallback 失败——这些是实际生产中最常见的问题，排查能力比编码能力更重要。
3. **动手做一个小的 Agent 场景**：不依赖 AI 辅助，自己从零实现一个多步推理 Agent（比如：用户输入一个业务需求 → AI 拆解为多个 SQL 分析子任务 → 逐个调用工具 → 汇总报告）。这个过程会帮你深刻理解 Agent 循环、工具编排、上下文管理。
4. **读 Spring AI 源码中的关键类**：`ChatClient`、`Advisor` 接口链、`ToolCallingManager` 这三个入口读一遍，大约 2-3 小时，收获远大于手写 CRUD。

### 4. 现实角度

转型 Spring + AI 方向，市场更看重的是：
- 你能不能**设计**一个可靠的 AI 应用架构
- 你能不能**评估和优化** RAG 效果
- 你能不能**排查和解决** AI 系统的生产问题
- 你能不能**选择合适的 AI 技术**来解决业务问题

而不是"这段代码是你手写的还是 AI 写的"。AI 辅助编程已经是行业共识，关键是你对整个系统有**掌控力**——知道为什么这样设计、出了问题知道怎么改。

---

**总结一句话**：你的两个项目已经覆盖了 AI 应用开发的核心三件套（LLM + RAG + MCP），下一步建议往 **Agent 多步推理** 和 **检索效果优化（Hybrid Search + Rerank）** 两个方向深入，这两个方向既有技术深度又有明显的业务价值。编码能力方面，重点放在"理解架构和数据流 + 排查问题能力"上，比手写每一行代码更有效。
